Executive Summary

1. Executive Summary

This guide provides a practical way for a client technical team to install and try SAQR Phase 1 locally using the preserved demo environment. It is designed for evaluation, walkthroughs, and limited hands-on testing of the user experience and core platform narrative.

Important Truth

The installation path in this guide is not a production deployment. It is a local evaluation path based on the preserved demo runtime. The production-ready handoff package exists for the delivery team and requires external prerequisites not provided in this repository.

Recommended Outcome Local Trial Validate the UI flow, demo runtime, and API heartbeat locally.
Technical Level Moderate Designed for technical reviewers comfortable with terminal commands.
Expected Time 20-30 Min Assuming Node.js is already installed.

Installation Steps

6. Step-by-Step Demo Evaluation Install

Step 1. Obtain the repository package

Place the SAQR repository on your machine by either:

  • cloning it from GitHub, or
  • extracting the provided ZIP into a local working folder.
git clone https://github.com/aboutony/SAQR.git
cd SAQR

Step 2. Open a terminal in the repository root

The repository root is the folder that contains package.json, apps, services, docs, and infra.

Step 3. Install all required dependencies

Run the Phase 1 bootstrap command from the repository root:

npm run phase1:bootstrap

Expected success signal: [phase1:bootstrap] workspace bootstrap complete

Step 4. Start the SAQR demo API

Open a dedicated terminal window and use the following PowerShell commands. These commands load the demo configuration into the current process and start the API in demo mode. 

Get-Content .env.demo.example | Where-Object { $_ -and $_ -notmatch '^\s*#' } | ForEach-Object {
    $name, $value = $_ -split '=', 2
    [System.Environment]::SetEnvironmentVariable($name, $value, 'Process')
}

Set-Location apps\api
npm start

Keep this terminal window open while you are evaluating SAQR.

Step 5. Verify the API heartbeat

In a browser, open the following URL:

http://127.0.0.1:3001/api/v1/sources/heartbeat

You should receive a JSON response with demo runtime status and the demo authority list.

Step 6. Launch the Shield UI

The fastest path is to open the Gateway Selector directly in your browser:

apps\shield-ui\GatewaySelector.html

If your browser or security policy does not allow local HTML execution comfortably, use a simple local static server from the apps/shield-ui folder: 

cd apps\shield-ui
python -m http.server 8080

Then open:

http://127.0.0.1:8080/GatewaySelector.html
First Run

7. First-Run Validation

Checkpoint What Success Looks Like
Bootstrap complete The install command completes without dependency errors.
API heartbeat The heartbeat URL returns JSON with "runtimeMode":"demo".
Gateway Selector opens The initial SAQR gateway page renders in your browser.
Dashboard opens After session setup, the Shield dashboard loads without a fatal error page.
Demo behavior appears You can see demo metrics, demo stream behavior, and preserved controls.
Validated In This Repository

The local demo API startup and heartbeat path were verified from this repository state while preparing this guide.

Try SAQR

8. How To Try SAQR

8.1 Initialize a session

  1. Open the Gateway Selector page.
  2. Enter a corporate email and organization ID.
  3. Select a market, industry, and language.
  4. Enter the dashboard.

8.2 Suggested evaluation path

  • Review the top-level dashboard metrics and authority heartbeat.
  • Inspect the Meta-Audit Stream table and the Instruction Drift panel.
  • Try the preserved demo controls if they are visible in the demo runtime.
  • Observe how the session changes based on market and industry selection.
What you may see Why it appears
Demo metrics and populated cards The demo runtime intentionally preserves embedded data and simulation flows.
Authority heartbeat data The demo API exposes a local heartbeat endpoint on port 3001.
Read-only posture badges SAQR is intentionally designed as a read-only compliance interception product.
No real client data This guide does not connect any client DB, VMS, or third-party platform.
What This Trial Does Not Prove

A successful local evaluation does not prove production readiness in the client environment. It only proves that the preserved Phase 1 demo path can be installed and tried locally.

8.3 Optional note for advanced technical reviewers

The repository also includes a production-ready handoff package for the delivery team. That path is documented separately and depends on real infrastructure such as shadow PostgreSQL, Kafka, an identity provider, and VMS connectivity. It is not a self-service client install path.

Troubleshooting

9. Troubleshooting

Issue Likely Cause Suggested Fix
npm run phase1:bootstrap fails Node.js or npm is missing, outdated, or blocked by network policy. Install Node.js 22 and rerun the bootstrap command.
Heartbeat URL does not respond The API terminal is not running or port 3001 is in use. Restart the API terminal and confirm nothing else is bound to port 3001.
UI opens but looks incomplete Local file restrictions or blocked external assets in the browser. Use the optional Python static server path and reopen the UI through http://127.0.0.1:8080.
The dashboard opens but some live calls fail The demo runtime still attempts heartbeat visibility through the API. Confirm the API terminal is running and the heartbeat URL returns JSON.
You expected real client integrations This guide is limited to the preserved demo path. Escalate to the delivery team and use the delivery handoff package instead.
Stop and Cleanup

10. Stop and Cleanup

10.1 Stop the local run

  • Close the browser tabs.
  • Return to the API terminal window and press Ctrl + C.
  • If you started a Python static server, stop it with Ctrl + C as well.

10.2 Remove the local package if required

If you only needed a temporary evaluation copy, delete the extracted or cloned SAQR folder after the trial is complete.

Escalation Path

If the client wishes to move beyond this local evaluation and into delivery-led implementation, the correct next artifact is the delivery handoff package in docs/handoff/README.md.
